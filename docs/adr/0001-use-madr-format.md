# 1. Use MADR Format for Generated ADRs

Date: 2026-03-01

## Status

Accepted

## Context

cADR generates Architecture Decision Records automatically from code changes. The
generated output needs a consistent, well-known structure so that teams can read,
search, and version-control the records without additional tooling. Several ADR
formats exist: Michael Nygard's original lightweight template, Alexandrian pattern
form, Y-Statements, and MADR (Markdown Architectural Decision Records).

## Decision

We adopted MADR as the output format for all ADRs generated by cADR. MADR provides
a lean but complete structure -- title, status, context, decision, and consequences --
written in plain Markdown. It is the most widely adopted ADR template in the open-source
ecosystem and is maintained by the adr.github.io community.

## Consequences

### Positive
- Generated ADRs are immediately familiar to teams already using MADR.
- Plain Markdown keeps records readable without specialized viewers.
- The structured sections map cleanly to the fields an LLM can reliably produce.
- Broad community adoption means good tooling support (linters, renderers, indexes).

### Negative
- Teams with an existing non-MADR format must either adopt MADR or post-process output.
- The fixed structure may feel rigid for decisions that do not fit neatly into the template.
